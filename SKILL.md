# web-access Skill

## 前置檢查

在開始聯網操作前，先檢查 CDP 模式可用性：

```bash
node "${CLAUDE_SKILL_DIR}/scripts/check-deps.mjs"
```

**Node.js 22+** 必需（使用原生 WebSocket）。

按指令碼輸出處理：
- `exit 0` → 繼續
- `exit 2` → 需詢問使用者偏好，寫入 `${CLAUDE_SKILL_DIR}/config.env` 的 `WEB_ACCESS_BROWSER`
- `exit 1` → 按 stdout 錯誤資訊處理。若提示包含「Agent 處理順序」，按其步驟執行（如先用系統命令開啟瀏覽器後重跑），自動可解則不打擾使用者；仍失敗再向使用者求助

支援引數 `--browser <chrome|edge>` 表達本次臨時覆蓋（不寫 config.env）。

切換瀏覽器時，proxy 是長駐程式，需先 `pkill -f cdp-proxy.mjs` 再重跑 check-deps。

檢查透過後並必須在回覆中向用戶直接展示以下須知，再啟動 CDP Proxy 執行操作：

```
溫馨提示：部分站點對瀏覽器自動化操作檢測嚴格，存在賬號封禁風險。已內建防護措施但無法完全避免，Agent 繼續操作即視為接受。
```

## 瀏覽哲學

**像人一樣思考，兼顧高效與適應性的完成任務。**

執行任務時不會過度依賴固有印象所規劃的步驟，而是帶著目標進入，邊看邊判斷，遇到阻礙就解決，發現內容不夠就深入——全程圍繞「我要達成什麼」做決策。這個 skill 的所有行為都應遵循這個邏輯。

**① 拿到請求** — 先明確使用者要做什麼，定義成功標準：什麼算完成了？需要獲取什麼資訊、執行什麼操作、達到什麼結果？這是後續所有判斷的錨點。

**② 選擇起點** — 根據任務性質、平台特徵、達成條件，選一個最可能直達的方式作為第一步去驗證。一次成功當然最好；不成功則在③中調整。比如，需要操作頁面、需要登入態、已知靜態方式不可達的平台（小紅書、微信公眾號、X(Twitter)、Bilibili、知乎、Reddit、HackerNews 等）→ **先檢查 OpenCLI 是否有對應 adapter**（`opencli list` 可查、目前支援 150+ 站）：有對應命令就用 `opencli <site> <verb>` 一行解決、附帶 adapter 反爬處理；OpenCLI 無對應 adapter 時 → 才走 CDP

**③ 過程校驗** — 每一步的結果都是證據，不只是成功或失敗的二元訊號。用結果對照①的成功標準，更新你對目標的判斷：路徑在推進嗎？結果的整體面貌（質量、相關度、量級）是否指向目標可達？發現方向錯了立即調整，不在同一個方式上反覆重試——搜尋沒命中不等於"還沒找對方法"，也可能是"目標不存在"；API 報錯、頁面缺少預期元素、重試無改善，都是在告訴你該重新評估方向。遇到彈窗、登入牆等障礙，判斷它是否真的擋住了目標：擋住了就處理，沒擋住就繞過——內容可能已在頁面 DOM 中，互動只是展示手段。

**④ 完成判斷** — 對照定義的任務成功標準，確認任務完成後才停止，但也不要過度操作，不為了"完整"而浪費代價。

## 聯網工具選擇

- **確保資訊的真實性，一手資訊優於二手資訊**：搜尋引擎和聚合平台是資訊發現入口。當多次搜尋嘗試後沒有質的改進時，升級到更根本的獲取方式：定位一手來源（官網、官方平台、原始頁面）。

| 場景 | 工具 |
|------|------|
| 搜尋摘要或關鍵詞結果，發現資訊來源 | **WebSearch** |
| URL 已知，需要從頁面定向提取特定資訊 | **WebFetch**（拉取網頁內容，由小模型根據 prompt 提取，返回處理後結果） |
| URL 已知，需要原始 HTML 原始碼（meta、JSON-LD 等結構化欄位） | **curl** |
| **目標站在 OpenCLI 支援清單**（小紅書、微信公眾號、X / Twitter、Bilibili、知乎、Reddit、HackerNews、LinkedIn、Amazon、1688、12306、36kr 等 150+ 站） | **優先 `opencli <site> <verb>`**（一行命令、adapter 自帶反爬處理） |
| 非公開內容，或已知靜態層無效的平台、但 OpenCLI 無對應 adapter | **瀏覽器 CDP**（OpenCLI 的 fallback、直接跳過靜態層） |
| 需要登入態、互動操作，或需要像人一樣在瀏覽器內自由導航探索 | **瀏覽器 CDP** |

瀏覽器 CDP 不要求 URL 已知——可從任意入口出發，透過頁面內搜尋、點選、跳轉等方式找到目標內容。WebSearch、WebFetch、curl 均不處理登入態。

**Jina**（可選預處理層，可與 WebFetch/curl 組合使用，由於其特性可節省 tokens 消耗，請積極在任務合適時組合使用）：第三方網路服務，可將網頁轉為 Markdown，大幅節省 token 但可能有資訊損耗。呼叫方式為 `r.jina.ai/example.com`（URL 前加字首，不保留原網址 http 字首），限 20 RPM。適合文章、部落格、檔案、PDF 等以正文為核心的頁面；對資料面板、商品頁等非文章結構頁面可能提取到錯誤區塊。

進入瀏覽器層後，`/eval` 就是你的眼睛和手：

- **看**：用 `/eval` 查詢 DOM，發現頁面上的連結、按鈕、表單、文本內容——相當於「看看這個頁面有什麼」
- **做**：用 `/click` 點選元素、`/scroll` 滾動載入、`/eval` 填表提交——像人一樣在頁面內自然導航
- **讀**：用 `/eval` 提取文字內容，判斷圖片/影片是否承載核心資訊——是則提取媒體 URL 定向讀取或 `/screenshot` 視覺識別

瀏覽網頁時，**先了解頁面結構，再決定下一步動作**。不需要提前規劃所有步驟。

### 補充：本地瀏覽器資源

使用者指向**本人訪問過的頁面**（"我之前看的那個講 X 的文章"、"上次開啟過的 XX 面板"）或**組織內部系統**（"我們的 XX 平台"、"公司那個 YY 系統"等公網搜不到的目標）時，檢索本地瀏覽器（Chrome / Edge）書籤/歷史：

```bash
node "${CLAUDE_SKILL_DIR}/scripts/find-url.mjs" [關鍵詞...] [--only bookmarks|history] [--browser chrome|edge] [--limit N] [--since 1d|7h|YYYY-MM-DD] [--sort recent|visits]
```

關鍵詞空格分詞、多詞 AND，匹配 title + url（可省略）；預設遍歷所有已安裝的 Chromium 系瀏覽器（Chrome、Edge），`--browser` 限定單一來源；`--since` / `--sort` 僅作用於歷史；預設按最近訪問倒序，`--sort visits` 按訪問次數排序（適合"高頻訪問的網站"這類場景）。

### 程式化操作與 GUI 互動

瀏覽器內操作頁面有兩種方式：

- **程式化方式**（構造 URL 直接導航、eval 操作 DOM）：成功時速度快、精確，但對網站來說不是正常使用者行為，可能觸發反爬機制。
- **GUI 互動**（點選按鈕、填寫輸入框、滾動瀏覽）：GUI 是為人設計的，網站不會限制正常的 UI 操作，確定性最高，但步驟多、速度慢。

根據對目標平台的瞭解來靈活選擇方式。GUI 互動也是程式化方式的有效探測——透過一次真實互動觀察站點的實際行為（URL 模式、必需引數、頁面跳轉邏輯），為後續程式化操作提供依據；同時當程式化方式受阻時，GUI 互動是可靠的兜底。

**站點內互動產生的連結是可靠的**：透過使用者視角中的可互動單元（卡片、條目、按鈕）進行的站點內互動，自然到達的 URL 天然攜帶平台所需的完整上下文。而手動構造的 URL 可能缺失隱式必要引數，導致被攔截、返回錯誤頁面、甚至觸發反爬。

## 瀏覽器 CDP 模式

透過 CDP Proxy 直連使用者日常瀏覽器（Chrome / Edge / Chromium 等 Chromium 系），天然攜帶登入態，無需啟動獨立瀏覽器。
若無使用者明確要求，不主動操作使用者已有 tab，所有操作都在自己建立的後臺 tab 中進行，保持對使用者環境的最小侵入。不關閉使用者 tab 的前提下，完成任務後關閉自己建立的 tab，保持環境整潔。

### 啟動

```bash
node "${CLAUDE_SKILL_DIR}/scripts/check-deps.mjs"
```

指令碼會依次檢查 Node.js、瀏覽器除錯埠，並確保 Proxy 已連線（未執行則自動啟動並等待）。Proxy 啟動後持續執行。

### Proxy API

所有操作透過 curl 呼叫 HTTP API：

```bash
# 列出使用者已開啟的 tab
curl -s http://localhost:3456/targets

# 建立新後臺 tab（自動等待載入）— URL 走 POST body，避免目標 URL 含 query 時被切分
curl -s -X POST --data-raw 'https://example.com' http://localhost:3456/new

# 頁面資訊
curl -s "http://localhost:3456/info?target=ID"

# 執行任意 JS：可讀寫 DOM、提取資料、操控元素、觸發狀態變更、提交表單、呼叫內部方法
curl -s -X POST "http://localhost:3456/eval?target=ID" -d 'document.title'

# 捕獲頁面渲染狀態（含影片當前幀）
curl -s "http://localhost:3456/screenshot?target=ID&file=/tmp/shot.png"

# 導航（URL 走 POST body，target 走 query）、後退
curl -s -X POST --data-raw 'https://example.com' "http://localhost:3456/navigate?target=ID"
curl -s "http://localhost:3456/back?target=ID"

# 點選（POST body 為 CSS 選擇器）— JS el.click()，簡單快速，覆蓋大多數場景
curl -s -X POST "http://localhost:3456/click?target=ID" -d 'button.submit'

# 真實滑鼠點選 — CDP Input.dispatchMouseEvent，算使用者手勢，能觸發檔案對話方塊
curl -s -X POST "http://localhost:3456/clickAt?target=ID" -d 'button.upload'

# 檔案上傳 — 直接設定 file input 的本地檔案路徑，繞過檔案對話方塊
curl -s -X POST "http://localhost:3456/setFiles?target=ID" -d '{"selector":"input[type=file]","files":["/path/to/file.png"]}'

# 滾動（觸發懶載入）
curl -s "http://localhost:3456/scroll?target=ID&y=3000"
curl -s "http://localhost:3456/scroll?target=ID&direction=bottom"

# 關閉 tab
curl -s "http://localhost:3456/close?target=ID"
```

### 頁面內導航

兩種方式開啟頁面內的連結：

- **`/click`**：在當前 tab 內直接點選使用者視角中的可互動單元，簡單直接，序列處理。適合需要在同一頁面內連續操作的場景，如點選展開、翻頁、進入詳情等。
- **`/new` + 完整 URL**：使用目標連結的完整地址（包含所有URL引數），在新 tab 中開啟。適合需要同時訪問多個頁面的場景。

很多網站的連結包含會話相關的引數（如 token），這些引數是正常訪問所必需的。提取 URL 時應保留完整地址，不要裁剪或省略引數。URL 透過 POST body 原樣傳入 `/new` 或 `/navigate`。

> **v2.5.3 遷移提示**：若引用的站點經驗檔案（`references/site-patterns/*.md`）或其它指令碼中仍含 `GET /new?url=...` 或 `/navigate?target=...&url=...` 的舊寫法，呼叫會收到遷移指引。按 [`references/migration-2.5.3.md`](references/migration-2.5.3.md) 就地改寫為 POST body 後再使用，並順手把該站點經驗檔案更新掉。

### 媒體資源提取

判斷內容在圖片裡時，用 `/eval` 從 DOM 直接拿圖片 URL，再定向讀取——比全頁截圖精準得多。

### 技術事實
- 頁面中存在大量已載入但未展示的內容——輪播中非當前幀的圖片、摺疊區塊的文字、懶載入佔位元素等，它們存在於 DOM 中但對使用者不可見。以資料結構（容器、屬性、節點關係）為單位思考，可以直接觸達這些內容。
- DOM 中存在選擇器不可跨越的邊界（Shadow DOM 的 `shadowRoot`、iframe 的 `contentDocument`等）。eval 遞迴遍歷可一次穿透所有層級，返回帶標籤的結構化內容，適合快速瞭解未知頁面的完整結構。
- `/scroll` 到底部會觸發懶載入，使未進入視口的圖片完成載入。提取圖片 URL 前若未滾動，部分圖片可能尚未載入。
- 拿到媒體資源 URL 後，公開資源可直接下載到本地後用讀取；需要登入態才可獲取的資源才需要在瀏覽器內 navigate + screenshot。
- 短時間內密集開啟大量頁面（如批次 `/new`）可能觸發網站的反爬風控。
- 平台返回的"內容不存在""頁面不見了"等提示不一定反映真實狀態，也可能是訪問方式的問題（如 URL 缺失必要引數、觸發反爬）而非內容本身的問題。

### 影片內容獲取

使用者瀏覽器真實渲染，截圖可捕獲當前影片幀。核心能力：透過 `/eval` 操控 `<video>` 元素（獲取時長、seek 到任意時間點、播放/暫停/全屏），配合 `/screenshot` 採幀，可對影片內容進行離散取樣分析。

### 登入判斷

使用者日常瀏覽器天然攜帶登入態，大多數常用網站已登入。

登入判斷的核心問題只有一個：**目標內容拿到了嗎？**

開啟頁面後先嚐試獲取目標內容。只有當確認**目標內容無法獲取**且判斷登入能解決時，才告知使用者：
> "當前頁面在未登入狀態下無法獲取[具體內容]，請在你的瀏覽器中登入 [網站名]，完成後告訴我繼續。"

登入完成後無需重啟任何東西，直接重新整理頁面繼續。

### 任務結束

用 `/close` 關閉自己建立的 tab，必須保留使用者原有的 tab 不受影響。

Proxy 持續執行，不建議主動停止——重啟後需要在瀏覽器中重新授權 CDP 連線。

## 並行調研：子 Agent 分治策略

任務包含多個**獨立**調研目標時（如同時調研 N 個專案、N 個來源），鼓勵合理分治給子 Agent 並行執行，而非主 Agent 序列處理。

**好處：**
- **速度**：多子 Agent 並行，總耗時約等於單個子任務時長
- **上下文保護**：抓取內容不進入主 Agent 上下文，主 Agent 只接收摘要，節省 token

**並行 CDP 操作**：每個子 Agent 在當前使用者瀏覽器例項中，自行建立所需的後臺 tab（`/new`），自行操作，任務結束自行關閉（`/close`）。所有子 Agent 共享一個瀏覽器、一個 Proxy，透過不同 targetId 操作不同 tab，無競態風險。

**子 Agent Prompt 寫法：目標導向，而非步驟指令**
- 必須在子 Agent prompt 中寫 `必須載入 web-access skill 並遵循指引` ，子 Agent 會自動載入 skill，無需在 prompt 中複製 skill 內容或指定路徑。
- 子 Agent 有自主判斷能力。主 Agent 的職責是說清楚**要什麼**，僅在必要與確信時限定**怎麼做**。過度指定步驟會剝奪子 Agent 的判斷空間，反而引入主 Agent 的假設錯誤。**避免 prompt 用詞對子 Agent 行為的暗示**：「搜尋xx」會把子 Agent 錨定到 WebSearch，而實際上有些反爬站點需要 CDP 直接訪問主站才能有效獲取內容。主 Agent 寫 prompt 時應描述目標（「獲取」「調研」「瞭解」），避免用暗示具體手段的動詞（「搜尋」「抓取」「爬取」）。

**分治判斷標準：**

| 適合分治 | 不適合分治 |
|----------|-----------|
| 目標相互獨立，結果互不依賴 | 目標有依賴關係，下一個需要上一個的結果 |
| 每個子任務量足夠大（多頁抓取、多輪搜尋） | 簡單單頁查詢，分治開銷大於收益 |
| 需要 CDP 瀏覽器或長時間執行的任務 | 幾次 WebSearch / Jina 就能完成的輕量查詢 |

## 資訊核實類任務

核實的目標是**一手來源**，而非更多的二手報道。多個媒體引用同一個錯誤會造成迴圈印證假象。

搜尋引擎和聚合平台是資訊發現入口，是**定位**資訊的工具，不可用於直接**證明**真偽。找到來源後，直接訪問讀取原文。同一原則適用於工具能力/用法的調研——官方檔案是一手來源，不確定時先查檔案或原始碼，不猜測。

| 資訊型別 | 一手來源 |
|----------|---------|
| 政策/法規 | 釋出機構官網 |
| 企業公告 | 公司官方新聞頁 |
| 學術宣告 | 原始論文/機構官網 |
| 工具能力/用法 | 官方檔案、原始碼 |

**找不到官網時**：權威媒體的原創報道（非轉載）可作為次級依據，但需向用戶說明："未找到官方原文，以下核實來自[媒體名]報道，存在轉述誤差可能。"單一來源時同樣向用戶宣告。

## 站點經驗

操作中積累的特定網站經驗，按域名儲存在 `references/site-patterns/` 下。

確定目標網站後，如果前置檢查輸出的 site-patterns 列表中有匹配的站點，必須讀取對應檔案獲取先驗知識（平台特徵、有效模式、已知陷阱）。經驗內容標註了發現日期，當作可能有效的提示而非保證——如果按經驗操作失敗，回退通用模式並更新經驗檔案。

CDP 操作成功完成後，如果發現了有必要記錄經驗的新站點或新模式（URL 結構、平台特徵、操作策略），主動寫入對應的站點經驗檔案。只寫經過驗證的事實，不寫未確認的猜測。

檔案格式：
```markdown
---
domain: example.com
aliases: [示例, Example]
updated: 2026-03-19
---
## 平台特徵
架構、反爬行為、登入需求、內容載入方式等事實

## 有效模式
已驗證的 URL 模式、操作策略、選擇器

## 已知陷阱
什麼會失敗以及為什麼
```
經驗/陷阱內容標註發現日期，當作"可能有效的提示"而非"保證正確的事實"。

## References 索引

| 檔案 | 何時載入 |
